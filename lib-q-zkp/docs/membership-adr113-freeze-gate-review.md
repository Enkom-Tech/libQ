## 1. Scope & Statement

This packet covers the soundness of the in-circuit (AIR/STARK) proof of the libQ unlinkable membership statement:

```text
∃ (secret t, path):
    MerkleVerify(root, L = H(t), path) = true
  ∧ N = H(domain ‖ t ‖ ctx)
reveal only (root, ctx, N);  L and t stay private.
```

- `H` = truncated-output Poseidon-256 wide sponge (state width 7, rate 2, capacity 5; digest = first 5 state cells) over `Complex<Mersenne31>` = GF(p²), p = 2³¹−1.
- `domain` is a **baked circuit constant** (`membership_domain()` from `MEMBERSHIP_DOMAIN_STR = "libq.zkfri.membership.v0"`), not a witness or public input.
- Public values = `[ root(5) ‖ ctx(2) ‖ N(5) ]` (12 elements). `L` and `t` stay private.
- Trace = one row per Merkle level; row 0 additionally carries the leaf and nullifier sponge blocks.

**Files audited:**
- `lib-q-zkp/src/air/unlinkable_membership.rs` — the membership AIR, trace gen, nullifier.
- `lib-q-zkp/src/air/wide_sponge.rs` — the single in-circuit Poseidon-256 sponge.
- `lib-q-zkp/src/air/poseidon_gadget.rs` — per-round Poseidon constraints (ARC / S-box / MDS).
- `lib-q-zkp/src/membership.rs` — public prove/verify API + path padding.
- `lib-q-zkp/src/merkle.rs` — `WidePoseidonMerkleTree`, `wide_node_hash`.
- `lib-q-zkp/src/lib.rs` — `ProofMetadata`, `merkle_tree_depth()`.
- `lib-q-stark-mersenne31/src/mersenne_31.rs` — field byte decode (`from_int` vs `from_canonical_checked`).
- `lib-q-zkp/src/air/mod.rs` — `merkle_root_from_bytes`, `poseidon_to_field`, `next_power_of_two`.

## 2. Confirmed Soundness Issues

**None.** After per-dimension finding generation, adversarial refutation (≥2-of-3 refuters), and independent lead-auditor re-reading of the actual code, **zero findings survived as exploitable circuit-soundness bugs**. No column or relation was found that a malicious prover could manipulate to produce a proof that VERIFIES while VIOLATING the decoded statement `MerkleVerify(verifier-supplied root, H(t), path) ∧ N = H(domain ‖ t ‖ ctx)`.

## 3. Per-Dimension Soundness Assessment

**Sponge underconstraint — SOUND.** Every round intermediate of every permutation is pinned by exactly one `assert_eq`: full rounds bind ARC (`poseidon_gadget.rs:305`), S-box `x⁵` (`:315`), and MDS (`:326`); partial rounds additionally copy-constrain the non-active lanes `local[col+i] == local[arc_col+i]` (`:363-368`) so they cannot be freely chosen. Column allocation advances by `n` per block with no aliasing (`columns_per_hash = (8+60)·(7·3) = 1428` per permutation). Capacity is carried as the threaded width-7 MDS *expression* between permutations (`wide_sponge.rs:96, :107`) with no free capacity column; the first ARC of each permutation binds all 7 incoming cells. The absorb + `10*1` padding schedule and `num_perms = ⌊L/2⌋+1` exactly match the value-level reference (`generate_wide_sponge_cells` / `poseidon256_wide_hash`), confirmed by `value_cells_digest_matches_reference_all_lengths` over lengths {0,1,2,3,5,7,10}. The 5-element digest is `state[..5]` of the genuine final-MDS expression, never a free read-back column. No exploitable underconstraint.

**Merkle threading — SOUND (with metadata caveat).** Per-fixed-height threading is correct: `dir` is boolean (`assert_bool`, `unlinkable_membership.rs:199`), the node input is direction-selected (`:203-212`), `next.running == parent` is gap-free over transition rows (`:230-233`), and the last row binds `parent == public root` (`:237-240`). The one real observation is that the **trace HEIGHT (proven level count) is never bound to the verifier's `tree_depth` metadata** — but this is not exploitable (see §4): the verifier always folds against its own trusted canonical root, so any height/depth relabel still reduces to honest "fold H(t) to the real root," giving a non-member no advantage.

**Leaf / same-t — SOUND.** `running[row 0]` is forced to `H(t)` via the row-0 binding `m_running[i] == leaf_out[i]` (`:276-278`) and then folded to the public root. There is exactly one committed `t` column slice (`T_START..T_START+3`), read once into `t_cols` (`:178-179`) and fed to BOTH the leaf sponge (`:249`) and the nullifier preimage (`:262`). Same-t is structural — there is no `t_leaf == t_null` constraint to omit because there is only one `t`. No decoupling possible.

**Nullifier integrity — SOUND.** The domain prefix is injected as constant expressions `AB::Expr::from(poseidon_to_field(&d))` for `d in membership_domain()` (`:259-261`) — no committed domain column exists anywhere in the row layout. The preimage is exactly `domain(2) ‖ t(3) ‖ ctx(2)` = 7 elements in the correct order, matching the value-level `membership_nullifier`. `ctx_cols` is bound to public ctx on row 0 (`:280-281`) and consumed by the same row-0 nullifier sponge; `null_out` is bound to public N on row 0 (`:284-286`). No malleability, no cross-input nullifier, no separate ctx/t witness for the nullifier.

**Ungated / padding rows — SOUND.** The leaf/nullifier sponges run ungated on every row (to keep max degree at 5; gating would raise degree to 6 and overflow the quotient blowup). The three row-0 bindings (`running == L`, `ctx == public ctx`, `N == public N`) are genuinely gated by `builder.when_first_row()` (`:273-287`). Rows 1.. carry valid zero-preimage hashes (`generate_membership_trace:356-360`) whose columns are read only from `current_slice`, are referenced by no binding, and cannot affect root, running-carry, N, or ctx. Fully isolated.

**Padding-root — SOUND (verifier-trusted root is the anchor).** `pad_root` (`membership.rs:176-183`) deterministically extends a root by `extra` zero-sibling `dir=left` levels; the arithmetic matches the honest fold with no off-by-one. The AIR is padding-blind — the pad levels' `dir`/`sibling` are free witnesses like any Merkle level. Soundness holds because the verifier feeds its OWN trusted canonical `root` argument into `pad_root` (`:300`); the prover cannot steer the target, and hitting it via a different top structure requires a Poseidon second-preimage/collision (infeasible at the intended security level). This is the standard, accepted Merkle-in-circuit soundness model.

**Public / metadata binding — SOUND at the field level (two hardening notes).** `root`, `ctx`, `N` are all bound to public values and nothing secret leaks. Two non-exploitable defects: (a) `tree_depth` is NOT cross-checked against the proof's STARK trace height `degree_bits` — the documented depth-confusion guard does not exist; (b) the byte-dispatch verifier decodes public values non-canonically. Neither lets a proof verify against a field-level statement it does not already attest to (see §4).

**Degree / FRI config — SOUND.** Max constraint degree is genuinely 5 (the ungated `x⁵` S-box; partial-round non-active lanes are degree-1 copies, full-round MDS is degree-1 in the sbox columns). No selector multiplies a degree-5 term (`when_first_row`/`when_transition`/`when_last_row` gate only the degree-1 bindings and the degree-1 carry). The quotient-chunk count is derived from the symbolic constraint AST by both prover and verifier, so any degree overflow fails loudly at prove time and can never silently verify.

## 4. Notable Dismissed Concerns (and why they are NOT bugs)

These were raised by the finders, survived to refutation in some cases, and were dismissed after exploit-construction analysis. They are documented so the cryptographer sees the reasoning rather than re-deriving it.

**(a) Depth / level-count malleability — `tree_depth` not bound to trace height.** *Mechanically true:* `verify_unlinkable_membership_with_config` (`membership.rs`) derives the checked target solely from prover-declared `tree_depth` (`pad_root(root, next_pow2(d)−d)`) and never asserts `1<<degree_bits == next_pow2(d)`; the STARK reads height from `proof.degree_bits` independently. *Not exploitable:* the target is always a deterministic image of the verifier's OWN trusted canonical root, so for any `(D, d)` relabel the prover must still fold `H(t)` to that honest target — exactly honest membership. A non-member would need a hash second-preimage. Level count / depth is NOT part of the public statement `[root ‖ ctx ‖ N]`. **Consequence:** the `tree_depth` doc-comment ("required for the depth-confusion guard") was **inaccurate** — no such guard existed, and proofs have non-unique declarable depth. This is a documentation/determinism defect, not a soundness hole. Addressed by O5.

**(b) Non-canonical byte decode of public values.** *True:* `merkle_root_from_bytes` (`air/mod.rs`) decodes each Mersenne31 word via `Mersenne31::from_int(u32::from_le_bytes(..))`, which **reduces mod p** rather than rejecting out-of-range words. So byte words `v` and `v+p` (for `v < 2³¹−1`) alias to the same field element; `wide_digest_from_bytes` / `ctx_from_bytes` / `verify_unlinkable_membership_bytes` inherit this. *Not a soundness bug:* both prover and verifier apply the same reduction, aliased byte-strings decode to the IDENTICAL field tuple, so the set of field statements a proof verifies against is unchanged — no field-level forgery. The honest serialize path is canonical (`as_canonical_u32`). The only conceivable harm is an *external* consumer that keys a nullifier/double-spend set on RAW serialized bytes without canonicalizing; no such in-repo sink exists. **Hardening:** switch the membership byte decoders to canonical-checked parsing. Addressed by O6.

**(c) ctx has no in-circuit range/canonical constraint.** Subsumed by (b): the row-0 binding `ctx_cols[i] == pub_ctx[i]` is a complete field-level equality; there is no underconstrained column. Any "out-of-range ctx" concern lives entirely in the deserializer, not the AIR.

**(d) Pad-levels unconstrained / `pad_root` width.** The pad levels having free `dir`/`sibling` is identical to the base wide Merkle path AIR and is the universally accepted Merkle-in-circuit model; soundness rests on Poseidon collision/preimage resistance, with the verifier-trusted canonical root as the anchor. No concrete forgery at the intended security level.

**(e) All four sponge sub-claims (round intermediates, capacity threading, schedule match, truncated output).** Each finder report self-classified as "Exploit: None". Independent re-reading confirmed each: every intermediate is pinned, capacity threads as an expression, the schedule matches the reference byte-for-byte, and the digest is the genuine final-MDS expression. No bug.

## 5. CRYPTOGRAPHER SIGN-OFF OBLIGATIONS (must be discharged before GREEN)

The automated/structural audit can only show that the *constraints faithfully encode* the intended hash and Merkle relations. The following are **load-bearing cryptographic and design questions that a human cryptographer MUST independently verify**. Until every item is signed off, this tier stays RED.

**O1 — Poseidon-256 round-count / algebraic-degree security over GF(p²). [LOAD-BEARING, BLOCKING FOR GREEN]**
The in-circuit and value-level Poseidon-256 use **8 full rounds + 60 partial rounds, `x⁵` S-box, state width 7**. These round counts are **NOT independently verified to be secure over the extension field `Complex<Mersenne31>` = GF(p²)**. The repeatedly-cited "~155-bit" figure is a **capacity-based collision target only** — it says nothing about whether 8+60 rounds resist algebraic attacks (Gröbner-basis / interpolation / higher-order differential / partial-round-specific attacks) **over GF(p²)**, where the algebraic structure differs from the base prime field the standard round-count formulas target. **Required:** a full round-count and algebraic-degree security review (including a Gröbner-basis cost estimate) for `x⁵`, width 7, 8+60 rounds over GF(p²), confirming ≥128-bit security or prescribing corrected round counts. **Every source file in scope carries an explicit `RED: Poseidon-256 round counts are NOT verified for GF(p²)` banner — this obligation is the reason the whole tier is gated.**

**O2 — Capacity-5 truncated-output collision bound.** Confirm that capacity 5 over ~62-bit GF(p²) elements genuinely delivers ≥128-bit collision resistance **for the truncated 5-element output sponge as used here** (absorb-rate-2, single padding permute, first-5-cell truncation). Validate that output truncation to 5 cells does not degrade the capacity-bounded collision/second-preimage resistance below 128 bits for the Merkle node hash, leaf hash, and nullifier hash.

**O3 — Domain-separation adequacy.** Confirm that the baked 2-element `domain = first 2 cells of H("libq.zkfri.membership.v0")` provides adequate cross-protocol / cross-statement nullifier separation, that the `domain ‖ t ‖ ctx` ordering yields a sound nullifier (no length-extension / absorption-boundary ambiguity given rate 2 and the `10*1` padding), and that there is no collision between the leaf preimage `t` (3 elems) and the nullifier preimage `domain‖t‖ctx` (7 elems) that could let one hash impersonate the other.

**O4 — Zero-knowledge. [hiding PCS + cryptographic blinding IMPLEMENTED 2026-06-22; only the formal simulator argument is BLOCKING]**
A hiding (zero-knowledge) prover path is wired: `membership::prove_unlinkable_membership_zk[_auto/_with_config]` uses `lib-q-stark`'s `HidingFriPcs` + salted `MerkleTreeHidingMmcs` (`is_zk()=1`; trace LDE'd at `log_blowup+1` with vanishing-polynomial randomization). The witness (`t`, leaf `L`, siblings, direction bits, all sponge intermediates) stays in the blinded trace; only `[root ‖ ctx ‖ N]` is public. Verified: round-trips through the public API; the verifier needs only the FRI parameters (not the prover's blinding seeds); fresh per-proof blinding entropy yields DISTINCT proof bytes; the nullifier is independent of blinding; the transparent verifier rejects ZK proofs (metadata `zk` flag). The degree-5 S-box forces `log_blowup >= 3` under the hiding PCS (its out-of-LDE extrapolation fallback is unimplemented), and the trace is padded to `MIN_ZK_DEPTH = 8` to clear the ZK FRI minimum.

**An adversarial ZK-hiding review (2026-06-22) confirmed two blinding-randomness leaks; BOTH are now FIXED in code:**
- **L1 — non-cryptographic RNG (was critical).** The hiding salts + blinding polynomials were generated by `lib_q_random::DeterministicRng`, a 64-bit xorshift64 self-documented "NOT cryptographically secure" (F2-linear; with salts exposed in the proof, de-blindable). **FIX:** new `lib_q_random::Kt128Rng` — a KangarooTwelve (SHA3-family XOF) CSPRNG with a **256-bit seed**, cryptographically pseudorandom and not linearly invertible. The `ZkConfig` MMCS/PCS now use `Kt128Rng`; all `zero_knowledge_tests` + membership ZK tests pass on the swap.
- **L2 — fixed/guessable seeds (was critical).** Convenience/default seeds were the constants `(0,1)` and the production API took only `[u64; 2]` (≤64-bit). **FIX:** `prove_unlinkable_membership_zk` now requires two **independent `[u8; 32]` CSPRNG seeds**, and `prove_unlinkable_membership_zk_auto` draws them from the OS CSPRNG internally (`lib_q_random::fill_entropy`) — it cannot be called with predictable/reused entropy and never falls back to weak seeds (errors if no OS entropy source).

**Still BLOCKING for GREEN (cryptographer, the one residual):** a formal ZK / **simulator argument** for this AIR + `HidingFriPcs`/`Kt128Rng` instantiation over GF(p²) — a simulator that, given only `(root, ctx, N)`, produces indistinguishable proofs — plus an independent re-confirmation that the L1/L2 fixes achieve the target hiding level and that the truncated-output sponge openings leak nothing under the OOD-opening vs randomization-degree budget. Production deployments MUST use `prove_unlinkable_membership_zk_auto` (or supply fresh CSPRNG `[u8;32]` seeds).

**O5 — Doc/spec defect: non-existent depth-confusion guard.** *(Addressed in code 2026-06-22 — see below.)* The `ProofMetadata::UnlinkableMembership.tree_depth` comment described a check that did not exist. Resolution: `verify_unlinkable_membership_with_config` now asserts `(1 << proof_trace_height_bits) == next_power_of_two(tree_depth)` so the declared depth is authenticated against the proof's actual trace height, and the doc comment is corrected. **Cryptographer to confirm** the cross-check is correct and that no downstream consumer trusts `merkle_tree_depth()` as an authenticated witness property beyond this.

**O6 — Canonical byte decode (hardening, LOW).** *(Addressed in code 2026-06-22 — see below.)* The membership byte decoders (`wide_digest_from_bytes`, `ctx_from_bytes`) now reject non-canonical words (`int >= 2³¹−1`) instead of silently reducing, so the byte encoding of the public statement is injective. **Cryptographer to confirm** the canonical-decode contract is sufficient for any external nullifier/double-spend set keyed on serialized bytes.

**O7 — Confirmed circuit findings.** None. There are no confirmed circuit-soundness findings to re-verify; this item is recorded as empty for completeness.

## 6. Verdict

**The circuit logic itself is sound, and there are NO blocking circuit bugs.** Across all eight audited dimensions — sponge underconstraint, Merkle threading, leaf/same-t, nullifier integrity, ungated/padding rows, padding-root, public/metadata binding, and degree/FRI config — no column or relation was found that a malicious prover could exploit to produce a proof that VERIFIES while VIOLATING the decoded statement. The surviving issues are a documentation defect (the non-existent depth-confusion guard, O5) and an encoding-hygiene defect (non-canonical byte decode, O6), both now addressed in code.

**However, this tier MUST remain RED.** The soundness conclusion is explicitly conditioned on cryptographic assumptions that no code audit can discharge — above all **O1 (Poseidon-256 round-count security over GF(p²) is unverified; the ~155-bit number is a capacity target, not a round-count/Gröbner-basis result)** and **O4 (the AIR is not yet argued zero-knowledge)**. **Verdict: RED — no blocking circuit-logic bugs; gated on human cryptographer sign-off (O1–O4 load-bearing).**

### O1 — Poseidon-256 round-count / algebraic-degree security over GF(p²)

**Instance.** `x⁵` S-box (α=5), state width `t = 7`, `R_F = 8` full + `R_P = 60` partial rounds
(68 total), over `GF(p²) = Complex<Mersenne31>`, `p = 2³¹−1`, so `log₂|F| = log₂(p²) ≈ 62`.
Target `M = 128`-bit security. (Code: `lib-q-poseidon::params::Poseidon256` width 7;
`constants::round_constants_256` = SHAKE256("Poseidon256_Mersenne31_v1_w7"), 476 constants.)

**Published round-count formulas** (POSEIDON, Grassi–Khovratovich–Rechberger–Roy–Schofnegger,
USENIX Security 2021 = IACR ePrint 2019/458, §5.5; HADES refinement ePrint 2023/537;
reproduced verbatim from the canonical generators `HorizenLabs/poseidon2`
`sat_inequiv_alpha` and `ingonyama-zk/poseidon-hash round_numbers.py`). With `n = log₂ p` (the
field-size term entering the algebraic bounds), `log₅2 = 0.43068`:

- **(a) Statistical (Eq. 2):** `R_F ≥ 6` if `M ≤ (⌊n − (α−1)/2⌋)(t+1)`, else `R_F ≥ 10`.
- **(b) Interpolation (Eq. 3–4):** `R_F + R_P ≥ 1 + ⌈log₅2 · min(M,n)⌉ + ⌈log₅ t⌉`.
- **(c) Gröbner (Eq. 5–6):** `R_F+R_P ≥ log₅2·min(M,n)`; `≥ (t−1) + log₅2·min(M/(t+1), n/2)`;
  `(t−1)(R_F+R_P) ≥ (t−2) + M/(2log₂α)`; plus the ePrint-2023/537 Macaulay-style `cost_gb4 ≥ M`.
- **Margin convention (§5.5):** add `+2` full rounds and `+7.5%` partial rounds to the minimum.

**Evaluation at α=5, t=7, M=128.**

| Regime | Statistical boundary | Binding bound (interpolation) | Margined optimal (R_F, R_P) | Deployed 8+60 |
|--------|----------------------|-------------------------------|-----------------------------|---------------|
| `n = 62` (realistic, GF(p²)) | 60·8 = 480 ≥ 128 ⇒ R_F ≥ 6 | R_F+R_P ≥ 1+⌈0.43068·62⌉+2 = **30** | (8, 26), total 34 | **sufficient, ≈2× total / 2.3× R_P** |
| `n = 31` (pessimistic subfield) | ⇒ R_F ≥ 6 | ≥ 1+⌈0.43068·31⌉+2 = **17** | (8, 14), total 22 | **sufficient, 4.3× R_P** |

`cost_gb4(8,60)` evaluates to **≈349 bits ≫ 128**. So 8+60 clears every published bound under
**both** the realistic `n=62` and the worst-case-subfield `n=31` substitution. **No round-count
increase is warranted; no parameter fix applied** (the task's "fix if insufficient" branch does
not trigger).

**Extension-field subtlety (decisive) + code checks performed.**
- `x⁵` IS a permutation of GF(p²): `gcd(5, p²−1) = 1` (`p ≡ 2 mod 5 ⇒ p²−1 ≡ 3 mod 5 ≠ 0`).
  Verified. (Also `gcd(5, p−1)=1`.)
- The only way GF(p²)'s tower structure could *reduce* security below the `n=62` reading is a
  **subfield descent**: if `GF(p)⁷` were invariant under the round function an attacker would work
  in the 31-bit subfield (`min(M,n)` → 31) and additionally unlock invariant-subspace / subspace-
  trail Gröbner attacks (Bariant et al. ToSC 2022:3; ePrint 2025/954). **Defense = generic round
  constants.** Code check: `generate_round_constants` (`constants.rs:110`) draws BOTH the real
  and the imaginary limb of every constant independently from SHAKE256 (`real = bytes[0..4]%P`,
  `imag = bytes[4..8]%P`), so all 476 constants have pseudorandom nonzero imaginary parts
  (`P[any imag = 0] ≈ 2⁻³¹`; a `debug_assert` also rejects any all-zero constant). Adding even one
  off-subfield constant per round maps `GF(p)⁷` off itself ⇒ **subfield invariance is broken; the
  `n=62` regime holds.**
- **Precise caveat for the reviewer:** the Cauchy MDS layer (`cauchy_mds`, `constants.rs`) is built
  from **base-field** nodes `Mersenne31::new(i+1)` / `Mersenne31::new(n+j+1)`, so its entries lie
  in `GF(p)`. The matrix is still MDS over GF(p²) (Cauchy ⇒ every minor nonsingular), but the
  **linear layer alone preserves `GF(p)⁷`**; the subfield defense rests **entirely on the round
  constants' genericity**, not the MDS. This is sound as-is, but a future change that derandomized
  the constants' imaginary parts would re-expose the subfield. Flag this dependency.

**Ground truth.** Plonky3 `poseidon2/round_numbers.rs` (M=128, α=5, R_F=8): Goldilocks `n=64`,
`t=8` → `R_P=27`; 31-bit `t=16` → `R_P=14`. Our `R_P=60` at `n≈62`, `t=7` is **>2× any deployed
comparable instance** — conservative, not thin.

**Adversarial self-check.** *Strongest "insufficient" case:* the 2024–2026 cryptanalysis wave
(EF Poseidon bounty; FreeLunch/CheapLunch ePrint 2024/347, 2025/2040; subspace-trail Gröbner
2025/954) lowers solving complexity below the Macaulay estimates baked into Eq. 5–6, and the
unusual choice of running the **state** (not just the FRI challenge field) over GF(p²) sits outside
the validated zkVM deployment envelope. *Resolution:* those attacks pressure `R_P` (they linearize
partial rounds), and `R_P=60` is 2.3–4.3× the requirement — the instances they render "borderline"
run `R_P` at ~1× the requirement; halving the effective `R_P` contribution still clears the bound.
The off-envelope concern reduces to the constant-genericity code check (performed, passes). The
case does not survive — but the residual on *future* cryptanalysis is real and is why human sign-off
remains required.

**Reviewer action items:** (1) confirm constant genericity argument (nonzero-imag) is the intended
subfield defense and is acceptable [code confirms it holds today]; (2) confirm the base-field MDS +
generic-constant composition has no residual invariant subspace; (3) confirm comfort with running
the permutation state over GF(p²) vs. the more common base-field-state convention.

### O2 — Capacity-5 truncated-output collision / second-preimage ≥ 128-bit over GF(p²)

**Instance.** Sponge over the width-7 GF(p²) permutation: **rate r = 2**, **capacity c = 5**,
absorb with `10*1` padding, digest = the **first 5 cells** of the final state (truncated output);
40-byte wire digest (`WIDE_DIGEST_ELEMS = 5`). Used for the leaf `L = H(t)`, every Merkle node
`H(left ‖ right)`, and the nullifier `N`.

**Derivation.** For a sponge built on a random permutation, the indifferentiability bound gives
collision and (second-)preimage resistance `min(2^{c·log₂|F| / 2}, 2^{w_out·log₂|F| / 2})` up to
the capacity, where `c·log₂|F|` is the capacity in bits and `w_out` the output width in cells
(Bertoni–Daemen–Peeters–Van Assche, *On the Indifferentiability of the Sponge Construction*,
EUROCRYPT 2008).
- Capacity bits: `c·log₂|F| = 5·62 = 310` ⇒ internal-collision resistance `2^{155} ≥ 2^{128}` ✓.
- Output bits: `w_out·log₂|F| = 5·62 = 310` ⇒ output birthday resistance `2^{155} ≥ 2^{128}` ✓,
  and the digest is `310 ≥ 256` bits, meeting libq-unlinkable-membership §9's ≥256-bit digest rule.
- Second-preimage / preimage ≥ min(output, capacity) bits = 155 ≥ 128 ✓.

So both the security-bearing axes (internal/capacity collision, and output collision) are `≈155`
bits — comfortably ≥128. The required capacity for `M=128` is `c ≥ 2M/log₂|F| = 256/62 ≈ 4.13`;
`c = 5` clears it.

**Truncation analysis.** The digest is a *truncation of the final state to 5 of 7 cells* (a
truncated-permutation output), not iterated rate-squeezing. Truncating to `w_out = 5` (≥ the
4.13-cell floor) preserves ≥128-bit output collision resistance; the 2 dropped cells only reduce
output entropy from 7→5 cells (434→310 bits), still ≥256. Crucially, *internal* collisions (the
ones that let an attacker equivocate a Merkle node) are governed by the **capacity (5 cells)**, not
the output width, so truncation does not weaken node-collision resistance.

**Adversarial self-check.** *Could the small rate (2) speed up an internal collision?* No — internal
collisions require colliding the full capacity (310 bits → 155-bit), independent of rate; a small
rate slows throughput, not security. *Could outputting exactly `c=5` cells (= capacity width) create
an algebraic relation between output and capacity?* The output cells are the *rate+part-of-capacity*
of the genuine final-MDS state (the audit confirmed the digest is `state[..5]` of the real final
permutation, never a free column), so they are a deterministic image of a full permutation with
hidden capacity remainder; no exploitable relation at the 128-bit level absent a permutation
distinguisher (which O1 governs). *Length-extension?* Sponges with nonzero capacity are not
length-extendable (unlike Merkle–Damgård).

**Dependency.** O2's "random permutation" premise is exactly O1 (the permutation must be a secure
PRP at the 128-bit level). O2 is sound *given* O1.

**PENDING HUMAN SIGN-OFF — tier stays RED.**

### O3 — Domain-separation adequacy (no leaf/nullifier preimage cross-collision)

**Construction.** `domain = first 2 cells of H("libq.zkfri.membership.v0")` (`membership_domain()`),
a **baked circuit constant** — not a witness, not a public input. Preimages:
`leaf = H(t)` over **3** cells `[t₀,t₁,t₂]`; `nullifier = H(domain ‖ t ‖ ctx)` over **7** cells
`[d₀,d₁,t₀,t₁,t₂,c₀,c₁]`. Same sponge, both with `10*1` padding.

**Claims and arguments.**
1. **Cross-protocol nullifier separation.** Because `domain` is a deterministic, fixed prefix
   derived from the unique statement string, any *other* protocol re-using this sponge with a
   different domain constant produces nullifiers in a disjoint image with overwhelming probability
   — a different 2-cell prefix forces a different absorption from round 1. This is the standard
   prefix-domain-separation argument; it holds as long as `domain` is collision-distinct from other
   protocols' constants (one K12/Poseidon evaluation of distinct strings; collision ≈ 2⁻¹⁵⁵).
2. **No leaf↔nullifier impersonation.** `leaf` and `nullifier` are computed by the same sponge but
   over inputs of **different length** (3 vs 7 cells ⇒ `num_perms = ⌊3/2⌋+1 = 2` vs `⌊7/2⌋+1 = 4`)
   and the nullifier carries the `domain` prefix the leaf lacks. For a leaf to be mistaken for a
   nullifier (or vice-versa) an attacker must find `t'` with `H(t') = H(domain ‖ t ‖ ctx)` — a
   **generic cross-length collision** (≈2⁻¹⁵⁵, O2), not a structural ambiguity, since the inputs are
   parsed unambiguously (next item).
3. **No absorption-boundary ambiguity / length-extension.** Rate-2 absorption with `10*1` padding
   makes every input uniquely decodable (the padding rule is injective on field-sequence inputs of
   any length), and the nonzero capacity precludes length-extension. There is no `(t, ctx)` and
   `(t', ctx')` of *different* shapes that absorb to the same padded block sequence.

**Adversarial self-check.** *Could a chosen `ctx` make the nullifier input `domain‖t‖ctx` equal some
other member's leaf input `t'`?* Lengths differ (7 vs 3) and the domain prefix is fixed and not
attacker-chosen, so this is again a generic collision, not a controllable structural collapse.
*Could `domain` accidentally collide with a `t`-prefix?* `domain` is 2 fixed cells from hashing the
statement string; a member's `t` is secret/independent — coincidental equality is ≈2⁻¹²⁴ and confers
no advantage (the attacker still cannot invert the sponge).

**Dependency.** O3 rests on O2 (collision/preimage resistance of the sponge) and O1 (the
permutation). O3 adds no new assumption beyond "the domain string is globally unique to this
statement," which is a registry/spec discipline (it is, per libq-unlinkable-membership §5/§9).

**PENDING HUMAN SIGN-OFF — tier stays RED.**

### O4 — Formal ZK / simulator argument (AIR + HidingFriPcs / Kt128Rng over GF(p²))

**What is implemented (not the residual).** `membership::prove_unlinkable_membership_zk[_auto/
_with_config]` uses `lib-q-stark`'s `HidingFriPcs` + salted `MerkleTreeHidingMmcs` (`is_zk()=1`):
the trace is LDE'd at `log_blowup+1` with vanishing-polynomial randomization, commitments are
salted, and FRI is run on the randomized codeword. Only `[root ‖ ctx ‖ N]` is public; the witness
(`t`, leaf `L`, siblings, direction bits, all sponge intermediates) stays in the blinded trace. The
**L1/L2 RNG leaks are fixed**: salts + blinding polynomials come from `lib_q_random::Kt128Rng` (a
KangarooTwelve XOF CSPRNG, 256-bit seed, not the F2-linear xorshift), and
`prove_unlinkable_membership_zk_auto` draws two independent `[u8;32]` seeds from the OS CSPRNG. The
nullifier is independent of blinding; fresh entropy ⇒ distinct proof bytes (tested); the verifier
needs only the public FRI params; the transparent verifier rejects ZK proofs by the `zk` metadata
flag.

**Simulator argument (the write-up that is the residual).** Goal: a PPT simulator `S` that, given
only `(root, ctx, N)` and the public params, outputs proofs computationally indistinguishable from
honest ones. Construction:
1. `S` builds a trace that is **consistent with the public values but blinded**: it runs the honest
   trace generator on an arbitrary dummy witness producing the *public* `(root, ctx, N)` is not
   possible without `t`; instead `S` relies on the **hiding PCS commitment being equivocable** —
   it commits to a random masking polynomial of the same degree budget and answers FRI/OOD queries
   using the standard zk-STARK simulation (Ben-Sasson–Chiesa–Riabzev–Spooner–Ward style + the
   hiding-PCS zero-knowledge of Haböck / the "ZK from low-degree randomization" line): the salted
   Merkle commitments are hiding, and the randomized codeword's openings at the (≤ `num_queries` +
   OOD) revealed points are distributed independently of the witness provided the randomization
   polynomial has degree > (#opened points).
2. Indistinguishability reduces to (i) the hiding/binding of `MerkleTreeHidingMmcs` salted
   commitments (each leaf salted with fresh `Kt128Rng` output ⇒ commitments reveal nothing), and
   (ii) the masking-degree budget exceeding the total number of query+OOD openings so that every
   revealed evaluation is uniform/independent of the witness.

**The quantitative residual a cryptographer must close:** confirm the **randomization-degree budget
vs. opening count**. With the degree-5 S-box forcing `log_blowup ≥ 3` and `ZK_NUM_QUERIES = 100`
(plus OOD points), the vanishing-poly/blinding degree added at `log_blowup+1` must strictly exceed
the number of openings the verifier learns, or a residual `O(1)`-bit leak per over-opened point
exists. The implementation pads the trace to `MIN_ZK_DEPTH = 8` and LDEs at `log_blowup+1`; the
reviewer must verify that the concrete masking degree under `HidingFriPcs` ≥ `#queries + #OOD + 1`
for the production params (and that the truncated-output sponge openings — 5 of 7 state cells — leak
nothing beyond the public digest under that budget).

**Adversarial self-check.** *Does revealing 5 of 7 sponge state cells (the truncated digest) leak
the 2 hidden capacity cells or the preimage?* The digest cells are public *as the statement*
already (`root`, `N`); the *intermediate* sponge cells live in the blinded trace and are only ever
opened at FRI query points, where the masking makes them uniform — so no, *given* the degree budget
holds. *Could the `Kt128Rng` seeding leak across proofs?* `_auto` draws independent OS-CSPRNG seeds
per proof; reuse would void hiding (documented, enforced by the `[u8;32]` API). *Is the AIR itself
zero-knowledge without the PCS?* No — a plain STARK trace is not ZK; the hiding PCS is load-bearing,
which is why this obligation is about the **PCS + AIR composition**, not the AIR alone.

**PENDING HUMAN SIGN-OFF — tier stays RED.** O4 is the residual that most directly blocks any
production *privacy* claim; the L1/L2 fixes are necessary but the formal simulator + degree-budget
confirmation are not self-certifiable.
